SMF Package Software Development Kit

Welcome to the world of modding!

This software development kit, for package and mod developers, is here to help people quickly begin to create packages and modifications for SMF. More thorough and advanced information can always be found on Simple Machines' Customize section on the Community Forums.

You can find the following information in this file:

For more information, you can always post on the forum asking questions, exchanging ideas and comments, or just plain bragging about how much better your package is than anyone else's.

What's available in this folder?

You should notice a few files that came with this readme. They are all examples to help give you a better understanding of how to create a package. They are as follows:
add_settings.php
An example of how to add settings to the database in a package. This file can be used by the package manager, and outside it.
boardmod.mod
A boardmod format modification file example. Note that this format's use is deprecated and should thus be avoided.
modification.xml
This is an example of a more advanced, xml based format for creating modification files.
package-info.xml
An example package-info manifest file. (it describes what files are in a package and what to do with them.)
package-info_advanced.xml
A much more advanced example of a package-info manifest, utilizing the features of the package system.
readme.html
This file, describing the use of the above files and what you can do with them.
All of these files, with the exception of the one you are currently viewing, are best viewed and edited with the text editor of your choice. If you can, try to read through them all and get a feel for what they are doing.

Creating a basic modification

One of the most popular types of package is a modification or mod. This is a package that modifies files in SMF so they work differently - adding, removing, or changing functionality. A large part of the package manager's advanced functions are built just to accommodate modifications.

However, if you're reading this... chances are you just want a more casual description. For this, open the modification.xml file included in this kit. Inside, you will see a basic example framework for modifying a specific file. A simplified version is as follows: (blue is only for comments.)
This tells it what file to edit. Among others, you can use $sourcedir, $themedir, and $languagedir.
<file name="$sourcedir/Subs.php">

   <operation>

      The position attribute should tell the position relative to the code you want to add.
      <search position="before"><![CDATA[
Search for some code to add stuff after.
]]></search>

      The code to add in the described location.
      <add><![CDATA[
This is what to add.
]]></add>

   </operation>

</file>
You can use the file part (element) as many times as you have files to modify. Then, inside that you can have as many operations as you want. In fact, even in the operations you can have more than one search - for example, one for before and one for after - just to make sure you find the right part of the file. (which is very important.)

On that note, try to make sure all the searches you make won't match any place but where you want them to. And, always test your modification before giving it out.... you never know if a typo might do you in!

You may notice the <![CDATA[...]]> used there. This simply makes it so you can use <, >, ", &, and etc., without using &amp; or something. And, the letters "CDATA" stand for "character data", meaning that it takes what's inside there literally - not as html/xml. The only restriction is that you cannot have ]]> inside it (because that's how it closes.)

Once you have the modification file sorted out, you need to modify the package-info file to make use of it. Open up package-info.xml, and make sure the <type> part says modification. Then make <install> look something like this: (not the blue parts!)
<install>
   Change this to the name of your readme file - if you have any.
   <readme>readme.txt</readme>
   Whatever filename you gave your modification file.
   <modification>modification.xml</modification>
   Add this only if you need to add settings - see add_settings.php. (most likely not.)
   <code>add_settings.php</code>
</install>
With that done, you're ready to package, test, and distribute your package.

Creating a package of avatars

Okay, so you have a bunch of avatars, a lot of graphical talent, and the need to let everyone see your avatars. No problem, we've got it covered ;).

All you need to do is open up package_info.xml, and change a few things. First, change the <type>modification</type> part to say 'avatar' instead of 'modification'. Next, find the big long part that starts with <install> and ends with </install>, and replace it with something like this:
<install>

   <readme>readme.txt</readme>

   <require-dir name="Cool_Avatars!" destination="$avatardir" />

</install>
If you don't have/want a readme, take out the whole readme line. Change Cool_Avatars! to whatever directory name you want to have your avatars in... (it's best to use a separate directory...) and bam, you're ready to go! All you need now is to read the stuff on packaging it all up.

Packaging together some language files

Translating is a difficult and time consuming job, and for this reason making packages for the finished product should, by all rights, be easy. Luckily, it is!

Just like with avatars and modifications, you will need to edit package_info.xml to change a few things. First, change the <type>modification</type> part to say 'language' instead of 'modification'. Next, find the big long part that starts with <install> and ends with </install>, and replace it with something like this:
<install>

   <readme>readme.txt</readme>

   <require-dir name="languages" destination="$themedir" />
   <require-dir name="langname" destination="$imagesdir" />

</install>
This assumes that the "languages" folder includes the language files themselves, and the "langname" folder has all the language specific images in it. You don't have to have a readme, so if you don't want one just remove that line entirely.
Now all that's left is packaging it all up.

Putting the whole thing together

Now that you have all your files ready, there are just a few things left to do. First, we're going to edit package-info.xml just a little more, to include information about the package, which makes it so everyone knows the name, version, and etc. for upgrade and support reasons.

Near the top, you should see three elements (tags): id, version, and name. Name is easy, just change it to the name of the package.... version should probably be 1.0 or similar. The id should be a specific format - your username, a colon, and then the package name. Ideally, the id should have no spaces in it. (don't worry, the package-info.xml file has more details.)

With that done, you'll want to zip it up. You can compress it with either the zip or tar.gz formats. The second is less common, but compresses more effectively. Either way you do it, compress the files into one archive - and you've got a package!

And remember - testing is everything. It's easy to make a little typo that will mess the whole thing up, so try it on a clean install of SMF before you give it out!

A last note on paths

As a last note, it's also very important, when writing modifications, not to use "Sources/Filename". It is a common practice to store those files outside of public_html, and so you should use the special package manager paths: $sourcedir/filename, $boarddir/filename, etc. Others available are $themedir, $languagedir, $avatardir, and $smileysdir.
